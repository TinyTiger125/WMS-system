# codex.md

## 项目背景与目标
- 底座：Odoo 19 Community（Sources）
- 场景：跨境轻量进销存（日本采购/入库，中国销售/分销）
- 当前边界：先完成进销存主链路与角色化工作台；工单系统后置
- 研发原则：不改 Odoo 核心源码，全部定制放 `custom/addons`

## 一、从零搭建阶段的关键问题与经验

### 1. Odoo 原生“应用商城感”过强，业务入口不清晰
- 现象：用户找不到“货品登记/商品档案”入口，操作动线靠猜。
- 根因：默认菜单按模块维度组织，不按业务流程组织。
- 处理：重构菜单为“商品主数据 -> 采购执行 -> 销售执行 -> 库存监控 -> 异常中心 -> 经营报表”。
- 经验：进销存产品必须先定义“业务动线菜单”，再讨论字段细节。

### 2. 中文化不彻底（页面混合中英文）
- 现象：导航中文，但局部字段/按钮/看板英文。
- 根因：Odoo 语言包与自定义文案混杂，且用户语言/公司语言未统一。
- 处理：
  - 安装并激活 `zh_CN`
  - 在初始化脚本中强制语言：内部用户 + 关联联系人 + 公司联系人统一为 `zh_CN`
  - 自定义业务字段与流程提示全部中文化
- 经验：中文化要在“部署初始化脚本”固化，而不是只靠手工后台设置。

### 3. 角色理解混乱（采购 vs 库管 vs 销售）
- 现象：管理员能看全部按钮，导致角色边界错觉。
- 根因：演示账号权限过高，且页面入口未按角色聚合。
- 处理：建立角色组（老板/库管/销售/客服）和角色仪表盘，按角色显示入口。
- 经验：演示阶段必须用“真实角色账号”走流程，不能用管理员感知体验。

## 二、开发中出现的高频技术错误与修复

### 1. OwlError: 字段未定义（`field is undefined`）
- 典型报错：
  - `"product.template"."name_jp" field is undefined`
  - `"product.template"."wms_min_qty" field is undefined`
- 根因：视图引用了不存在字段，或模块未升级导致数据库元数据未刷新。
- 修复：
  - 检查模型字段定义与视图一致性
  - 执行模块升级 `-u custom_wms_base,...`
  - 清理浏览器缓存后重试
- 预防：每次新增/改名字段后，先跑模块升级再做 UI 验证。

### 2. 视图加载顺序导致 External ID not found
- 典型报错：
  - `External ID not found: custom_wms_base.action_custom_wms_user_quick_create`
- 根因：`res_users_views.xml` 先引用动作，但动作定义在后加载文件。
- 修复：在 `__manifest__.py` 中调整 data 顺序，先加载动作定义文件，再加载引用视图。
- 预防：凡 `%(module.action_xxx)d` 的按钮，动作 XML 必须先于视图加载。

### 3. 安装/升级脚本模块清单不一致
- 现象：本地功能正常，服务器缺功能（如 `custom_cn_sync` 未装）。
- 根因：`compose/.env.example/init-db.sh` 的模块列表不一致。
- 修复：统一三处配置：
  - `ODOO_INIT_MODULES`
  - `ODOO_UPGRADE_MODULES`
  - 含 `custom_cn_sync`
- 预防：建立“单一真源”配置模板，并做发布前一致性检查。

### 4. 按钮行为不一致导致学习成本高
- 现象：不同页面“返回/取消”位置与行为不统一。
- 修复：统一规则：
  - 新建未保存：`取消并返回`
  - 已保存：`返回列表`
  - 统一在 form header 固定位置
- 经验：一致性比单点“炫功能”更重要。

### 5. 权限报错（读写跨模块字段）
- 典型场景：销售单读取发票行、异常中心/日报表访问被拒。
- 根因：角色组缺少必要的关联模型读取权限。
- 修复：补齐角色组最小必要权限与 record rule，跑全量角色扫描。
- 预防：每次改菜单/动作都要执行角色权限冒烟脚本。

## 三、业务流程认知统一（产品层经验）

### 1. 采购下单 vs 采购入库
- 采购下单：采购角色发起承诺（业务决策）。
- 采购入库：库管角色验货入账（库存事实）。
- 联动：采购单确认后生成待入库任务，库管验证后才增库存。

### 2. 销售订单 vs 销售出库
- 销售订单：销售角色承诺（锁库存/计划）。
- 销售出库：库管执行实物流转（库存扣减）。
- 联动：订单确认后进入待出库，库管复核并完成出库。

### 3. 老板角色是“采购+销售+看板”复合角色
- 经验：老板账号应可建商品、看采购与销售、看库存价值与异常，不应被误限权。

## 四、UI/UX 调整经验（易用性优先）

### 1. 导航结构
- 结论：顶部导航更符合当前产品整体视觉与 Odoo 框架约束。
- 经验：大改为左侧导航在当前主题下破坏感强，不建议强推。

### 2. 色彩方案
- 问题：默认紫+绿冲突明显，视觉不专业。
- 处理：统一主题主色与强调色，减少跳色。
- 经验：B2B业务系统优先“稳定、低刺激、可读性”。

### 3. 页面内流程提示
- 处理：关键单据页加入“当前阶段/建议下一步”提示文案。
- 经验：流程提示能显著降低培训成本。

## 五、部署与发布阶段问题

### 1. 本地稳定版 -> 服务器发布策略
- 约束：只把本地“稳定版本”推服务器，禁止在服务器直接热修逻辑。
- 经验：服务器仅承载已验证版本，问题回到本地修复后再发布。

### 2. 一键安装包与覆盖更新
- 做法：生成 `custom-wms-installer-1.0.0.tar.gz`，服务器解压覆盖代码，保留 `deploy/.env`。
- 注意：保留环境变量中的密码与端口，但要补齐模块/语言相关参数。

### 3. 上线验收标准（必须自动化）
- 容器健康：`wms_app`, `wms_db` 均 `healthy`
- HTTP健康：`/web/login` 返回 `200`
- 模块状态：核心模块 `installed`
- 中文状态：`zh_CN` active，活跃内部用户语言全中文

## 六、当前沉淀的工程规范（建议长期执行）

### 1. 开发规范
- 不改核心源码；仅改 `custom/addons`
- 所有字段改动必须伴随视图/权限/升级验证
- 功能变更必须补一条“角色影响说明”

### 2. 发布规范
- 发布前必跑：
  - 模块升级检查
  - 角色权限扫描
  - HTTP健康检查
- 通过后再打包，再发布服务器

### 3. 回归规范
- 至少覆盖四条主链路：
  - 商品登记
  - 采购下单->采购入库
  - 销售下单->销售出库
  - 经营报表/库存价值

### 4. 配置规范
- 部署脚本中固化：
  - `ODOO_FORCE_LANG=zh_CN`
  - 完整模块清单（含 `custom_cn_sync`）
- `.env` 与脚本参数必须一致

## 七、已知遗留与后续优化建议
- 英文残留可能出现在第三方未翻译字段（属于语言包范围，不是功能故障）。
- `--without-demo=all` 在 Odoo 19 会有警告（可改为布尔参数风格，属于清理项）。
- `_sql_constraints` 警告提示未来可升级为新约束声明写法（不影响当前可用性）。
- 未来可补：
  - 自动化 UI 冒烟（Playwright）
  - 备份/灾备脚本（数据库定时备份 + 恢复演练）
  - 上线后问题回收看板（版本化改进）

## 八、结论
这次迭代最大的经验不是“做了多少功能”，而是建立了可重复的交付方法：
- 先流程后字段
- 先角色后权限
- 先本地稳定后服务器发布
- 先验收标准后版本上线

按这套方法继续推进，后续版本的准确性和稳定性会显著提升。
